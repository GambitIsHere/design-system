What this prompt audits

SEO Route coverage, hreflang correctness, canonical tags, sitemap.xml presence, robots.txt, JSON-LD structured data, meta tags, Open Graph, Twitter Cards, heading hierarchy, programmatic SEO scalability.

Perf Lighthouse lab scores (Performance, SEO, A11y, Best Practices), Core Web Vitals lab values (LCP, CLS, INP, TBT, FCP, TTFB), CrUX field data (real-user 28-day p75 metrics).

Both Render-blocking resources, JS bundle size vs crawl budget, image optimisation, mobile responsiveness, HTTPS, viewport — these affect both scores.

What this prompt does NOT audit

Backlink profiles (requires Ahrefs/SEMrush). Keyword rankings (requires rank tracker). Competitor content depth (covered in the portfolio audit's competitor-alternatives skill). Repo code quality (portfolio audit). E-reputation / Trustpilot (portfolio audit). Content editorial quality (subjective, needs human review).

How it plugs into the portfolio audit

Each product's SEOScore and PerfScore feed directly into the A/B/C decision. A product with PerfScore < 40 and SEOScore < 50 on a legacy domain is a strong Option B candidate. A product with strong scores argues for Option A. Every finding gets a fixable-in-place vs architectural label — the handoff back to the strategic decision.

URL tiering per product

Instead of auditing every route × every locale (thousands of URLs, mostly redundant), use a 4-tier sample that captures the distinct signal in each category:

Form factor — mobile weighted higher than desktop

Every URL runs twice: strategy=mobile and strategy=desktop. Mobile weighted higher in PerfScore because Sanjow is >70% mobile. Desktop still captured — AOV tends to be higher on desktop for travel products, so desktop perf affects conversion even though traffic share is lower.

Route discovery

For repo-based products: parse src/app/ or src/pages/ for the Next.js route manifest. For live-only sites: parse sitemap.xml (top 50 URLs by priority) and do a shallow crawl (depth 2) to fill gaps. Never audit a URL that returns non-200 — log as a 404 finding instead.

Budget per product

4 tiers × 2 locales × 2 form factors = 16 PSI calls per product. For 14 products: ~224 PSI calls per portfolio run. Well under the 25,000/day quota. Raw JSON dumps saved to data/psi-raw/ for reproducibility.

Execution stages

1

Discover

Reuse repo list from portfolio audit. For each product: locate live domain(s), parse routes from repo and/or sitemap.xml, build the URL tier list. Produce domain-manifest.yml.

2

Fetch PSI + CrUX

PageSpeed Insights API for each URL (Lighthouse lab + CrUX field). 200ms delay, retry on 5xx with exponential backoff, wait 60s on 429. Log URLs where CrUX has insufficient data — that's itself a signal.

3

Static SEO audit

For each URL, parse raw HTML (no JS execution): title, meta description, canonical, hreflang, robots, JSON-LD, OG, Twitter Cards, h1 count, heading hierarchy. Fetch /robots.txt and /sitemap.xml once per domain.

4

Score

Apply weighted model from Scoring tab. Produce SEOScore + PerfScore per product. Weighted average across tiers (Tier 1 + 3 = 2×, Tier 2 + 4 = 1×). Show arithmetic — no opaque scores. Every number cites source URL and field.

5

Decide impact

Label every finding: Fixable in-place (supports Option A) or Architectural (supports Option B). This is the handoff back to the portfolio audit's A/B/C decision.

6

Write + commit

One markdown report per product in seo-pagespeed-audit/products/. One portfolio summary with the score matrix. Machine-readable CSV at data/scores.csv. Commit with conventional message.

Severity scale for findings

P0 blocker
Breaks indexing or loses money. Noindex on prod, canonical loops, CLS >0.25 in checkout, LCP >6s mobile homepage.
P1 high
Measurable ranking or conversion impact. Missing meta description, CLS 0.1–0.25, mobile LCP 3–4s, no Open Graph.
P2 medium
Compounds over time. Suboptimal h1 hierarchy, non-WebP images, thin content.
P3 low
Polish. Security headers missing, no HTTP/3, IPv6 off.
